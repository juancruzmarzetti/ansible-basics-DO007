Ansible Basics:
What makes up an Ansible playbook?
Playbook = Plays, Modules and Plugins.

What are they?
- Top level specification for a group of tasks.
- Will tell that play which hosts it will execute on and control behavior such as fact gathering or privilege level.

Building blocks for playbooks:
- Multiple plays can exist within an Ansible playbook.

Example (playbook.yml):

---
- name: Ensure the httpd package is installed
  hosts: web
  become: true

  tasks:
    - name: Ensure the httpd package is installed
       ansible.builtin.package:
         name: httpd
         state: present

    - name: Create the index.html file
      ansible.builtin.template:
        src: files/index.html
        dest: /var/www/html/

    - name: Start the httpd service if needed
      ansible.builtin.service:
        name: httpd
        state: started


other examle:
setup-playbook.yml:

---
- name: Basic System Setup
  hosts: web
  become: true
  tasks:
    - name: Update all security-related packages
      ansible.builtin.package:
        name: '*'
        state: latest
        security: true

    - name: Create a new user
      ansible.builtin.user:
        name: roger
        state: present
        create_home: true

The top level entry, which in this case, is named 'Basic System Setup', defines the manage nodes to target, here the main focus is getting the Basci System Setup on a group of hosts labeled 'web'.
The 'become' entry focuses on privilige escalation. For example, enabling sudo access to ensure you can install packages on the system which is necessary for system level changes.
And within each play, we have tasks, the individual steps that Ansible will follow to complete a specific action. 
Each task is paired with a module, with just like a pre-built function that performs the action.
Modules are what make Ansible flexible and powerful, covering everything from package management to user creation.
For example the ansible.builtin.package manages software packages, here we are ensuring that the security packages are installed by using the wildcard ('*') and setting security equal to true.
If we wanted to update all the packages on the system, we just would have to remove the 'security: true' line and it would have that effect.
The ansible.builtin.user is a module that creates a user to the target systems, in this case, it creates the roger user on the systems that belong to the web server group.

The ansible modules are parametrized components with internal logic, respresenting a single step to be done.
They are usually created in Python or Powershell for Windows setups, but can be developed in any language.

plugins are another piece of the ansible puzzle, they add functionality by adding features that extend what modules and plays can do.
While modules can handle specific tasks, plugins allow us to customize Ansible's behaviour in a variety of ways.
They handle things like privilege escalation (become), formatting outputs or managing connections.
In our example the 'became' plugin enables Ansible to run tasks with elevated privileges, like using sudo, which is essential for tasks like installing software or starting services.
The nice thing about plugins aside from the many builtin plugins you can use, is that you can also create custom plugins to suit your unique needs, making it highly addaptable.

Ansible inventory:
An Ansible inventory is a list of systems in your infrastructure that automation is executed against.
Example:
(hosts.ini)

[web]
webserver1.example.com
webserver2.example.com

[db]
dbserver1.example.com

[switches]
leaf01.internal.com
leaf02.internal.com

An inventory is a list of hosts or a list of groups of hosts that Ansible will manage.
Inventories can be as straightforward or as complex as you need, grouping hosts, basing on their roles, locations, or any other criteria.
This approach allows us to define groups once and re-use them in multiple playbooks.
Making it easy to run specific tasks on selected groups of servers or devices without specifying which servers we want each time.

Ansible also supports dynamic inventories:
Dynamic inventories automatically pull host information from external sources like CLoud providers, such as AWS or Azure, and virtualization platforms, like Openshift virtualization.
This means as your infrastructure scales or changes, Ansible can dynamically adjust the list of managed hosts, which means no need to manually update your inventory file.
This flexibility is especially powerful in environments where systems are frequently added or removed.
 

Ansible variables:
You can define variables in your playbooks, inventory, or variable files.

Example of defining variables on playbooks:

---
- name: Basic System Setup
  hosts: web
  become: true
  vars:
    user_name: 'roger'
    package_state: 'latest'
    security_only: true
  tasks:
    - name: Update all security-related packages
      ansible.builtin.package:
        name: '*'
        state: "{{ pachage_state }}"
        security: "{{ security_only }}"

    - name: Create a new user
      ansible.builtin.user:
        name: "{{ username }}"
        state: present
        create_home: true


Production server might need specific users created or different package settings.
Development servers might only need basic updates and users.
With variables we can adjust the settings for each case without rewrtigin the playbook.

Ansible Facts:
Just like variables, but coming from the host itself.
Ansible facts are built-in variables that that provide detailed information about the systems you're managing.
This facts include information like IP addresses, system architecture, operating system versions, network configurations, you name it. 

Example of Ansible Facts Playbook:

---
- name: Ansible Facts playbook
  hosts: localhost
  gather_facts: true

  tasks:
    - name: Define custom fact
      ansible.builtin.set_fact:
        server_role: "frontend"

    - name: Use custom and built-in fact
      ansible.builtin.debug:
        msg: 'The {{ ansible_hostname }} is assigned the role: {{ server_role }}'

As in this example, you can go in and create a simple playbook that captures those different hosts and the  displays those facts for you using a built-in ansible module like the debug module.
We then could specify the variable ansible_facts, which is a special variable that will you us all the different facts about the systems or devices we're automating against.

Ansible Constructs:
- Condicionals
- Handlers
- Loops

Theze tools are essential for tailoring your playbooks for specific environments and use cases.

Conditionals: allow tasks to run only if certain conditions are met.
Enable dynamic automation by checking the values of variables & making decisions at runtime.
Example:
---
- name: variable playbook test
  hosts: localhost

  vars:
    my_mood: happy

  tasks:
    - name: task, base on my_modd var
      ansible.builtin.debug:
        msg: "Yay! I am {{ my_modd }}!"
      when: my_mood == "happy"

    - name: task, based on my_mood var
      ansible.builtin.debug:
        msg: "Ask at your own risk. I'm {{ my_mood }}!"
      when: my_mood == "grumpy"

Ansible conditionals with facts:

---
- name: variable playbook test
  hosts: localhost

  tasks:
  - name: Install httpd
    ansible.builtin.package:
      name: httpd
      state: latest
    when: ansible_distribution == 'RedHat'

  - name: Install apache
    ansible.builtin.package:
      name: apache2
      state: latest
    when: ansible_distribution == 'Debian' or
          ansible_distribution == 'Ubuntu'


You also can use also use the results of a previous task to decide wether to execute a busequent task:

---
- name: variable playbook test
  hosts: localhost

  tasks:
  - name: Ensure httpd pachages are present
    ansible.builtin.package:
      name: httpd
      state: latest
    register: httpd_results

  - name: Restart httpd
    ansible.builtin.service:
      name: httpd
      state: restarted
    when: httpd_results.changed


Whe do it by using the 'register' directive to store the result of the first task, and then if that result had a change, the 'when' condition found within the second task would become true.


Ansible Handlers:
Handlers are special tasks that are triggered by other tasks by using the 'notify' directive.
They are typically used for actions like restarting services or reloading configurations, and they only run when something has changed.

Example:
---
- name: variable playbook test
  hosts: localhost

  tasks:
  - name: Ensure httpd package is present
    ansible.builtin.package:
      name: httpd
      state: latest
  notify: restart_httpd

  handlers:
  - name: restart_httpd
    ansible.builtin.service:
      name: httpd
      state: restarted


Another key about handlers is they always run at the end of your playbook.

Ansible Templates:
Templates are files that dynamically generate content by using variables, conditionals, loops and filters.
How do they work?
- First by reading in the source template file (.j2 (Jinja2))
- Filling in the dynamic values based on the variables or facts.
- Deploy the processed file with the variable values in place.

For example, configuring a web server like apache may require dynamic values for the ports it listens on or the document route for the hosted files.
This is where ansible template module (ansible.builtin.template) comes into play. It processes the template file and deploys the customized configuration to the managed nodes or your devices.

Example:

---
- name: Ensure apache is installed and started
  hosts: web
  become: true
  vars:
    http_port: 80
    http_docroot: /var/www/misyte.com
  
  tasks:
    - name: Verify correct config file is present
      ansible.builtin.template:
        src: templates/httpd.conf.j2
        dest: /etc/httpd/conf/httpd.conf

These two variables in this example playbook are going to allow us to actually modify what's actually being configured in that file dynamically for our web server hosts.

Example of a .j2 file:

## Excerpt from httpd.conf.j2

# Chnage this to listen on specific IP addresses as shown below to prevent Apache from glomming onto all bound IP addresses.
#
# Listen 80  ## original line
Listen {{ http_port }}

# DocumentRoot: The directory out of which you will serve your documents.
# DocumentRoor "/var/www/html"
DocumentRoot {{ http_docroot }}


So in this template we're doing that dynamic configuration.


Ansible Roles:
- A method to modularize the different components of your automation.
- Think of them as re-packaged instructions that group tasks, variables, files, templates into reusable components.

Key Components:
- Tasks
- Variables
- Files
- Templates
- Handlers

Here's how they break down:
- Tasks: the core of your automation, defined stpe-by-step in tasks/main.yml.
- Variables: configurations you can adjust, stored in vars/ or defaults/.
- Files: static resources like scripts or configurations you need to deploy, stored in the files/ directory.
- Templates: dynamic files that can be customized with variables, located in the templates/ directory.
- Handlers: actions triggered by task changes, such as restarting a service, saved in handlers/main.yml.

This structured approach makes it easier to organize your playbooks and share automation across your team or projects.

Ansible roles provide:
- Reusability: write once, reuse multiple times.
- Maintainability: clear structure for easy updates.
- Collaboration: share with teams for consistency.
- Scalability: modular design for growing projects.

How to create an Ansible Collection?
In our case we want to create a collection of our apache web servers ansible configuration, so, we run the following command line below:

ansible-galaxy collection init webops.apache --init-path ./collections/ansible_collections  

(the 'webops' part is the namespace, it's what differentiates the apache collection of different users, in this case, it's the 'apache' collection of 'webops', also that namespace should be unique).
(then the '--init-path ./collections/ansible_collections' part of the command line creates the specified collection inside the directory 'ansible_collections' and also creates this directory inside a another new directory named 'collections')
(inside the 'ansible_collections' directory, this command line creates the basic structure of a collection, firstly the directory 'webops' and inside that creates the directory 'apache', which will have the tree pre-format which we specified 
previously).


from our role-playbook.yml, now we can refer to the apache role by two different ways:

---
- name: Deploy Apache using a Role
  hosts: web
  become: true
  roles:
    - webops.apache.apache

or

---
- name: Deploy Apache using a Role
  hosts: web
  become: true
  collections:
    - webops.apache
  roles:
    - apache


When we work with collections and we don't have downloaded it yet, and we want to avoid all mistakes when executing the ansible configuration, we should create a requirements.yml gile in which we would
specify the names of the collections that we gonna use in our ansible configuration.

Role-based Acces Control (RBAC): (Ansible Automation Platform functionality)
Securely govern access to your automation:
- Logically group controller objects and grant users and teams read, execute, edit permissions.
- Use predefined roles to gran access.
- Integrates with your existing entreprise authentication systems.

This means you can define exactly what users or teams can do down to specific job templates, inventories, or credentials.
For example, you might allow one user to execute a job, another user to edit, and a team to manage only specific job templates.

User Management:
- An organization is a logical collection of users, teams, projects, inventories and more. All entities belong to an organization.
- Teams provide a means to implement role-based access control schemes and delegate responsibilities across organizations.
- A user is an account to access Ansible Automation Controller and its services given the permissions granted to it.
